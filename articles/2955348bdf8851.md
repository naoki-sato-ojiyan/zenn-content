## はじめに

プログラミング完全初心者がMacでWeb開発環境を構築し、Django+Docker+PostgreSQLでToDoアプリを作った記録です。
同じように初心者の方の参考になれば嬉しいです。

## 作ったもの

- **ToDoアプリ**（追加・削除・完了切り替え）
- **ユーザー認証**（ログイン・ログアウト）
- **ユーザーごとのTodo管理**
- **Docker + PostgreSQL**による本番環境に近い構成

## 使用技術

- Python 3.14
- Django 6.0
- PostgreSQL 16
- Docker / Docker Compose
- Bootstrap 5
- GitHub

## 第1章：環境構築

### Homebrewとは？

Macのアプリやツールをコマンドでインストールできるパッケージマネージャーです。
開発者はほぼ全員使っています。

```
【App Store】
クリック → 検索 → インストール → 完了

【Homebrew】
ターミナルで brew install python → 完了
```

| 操作 | コマンド |
|---|---|
| インストール | `brew install パッケージ名` |
| アンインストール | `brew uninstall パッケージ名` |
| アップデート | `brew upgrade` |
| バージョン確認 | `brew --version` |

### Homebrewのインストール

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

インストール後はPATHの設定が必要です。

```bash
echo >> ~/.zprofile
echo 'eval "$(/opt/homebrew/bin/brew shellenv zsh)"' >> ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv zsh)"

# 確認
brew --version
```

### Pythonのインストール

Mac標準のPythonではなくHomebrew版をインストールします。

```bash
brew install python@3.13

# PATHを通す
echo 'export PATH="/opt/homebrew/bin:$PATH"' >> ~/.zprofile
source ~/.zprofile

# 確認（/opt/homebrew/bin/python3 と表示されればOK）
which python3
python3 --version
```

:::message
`which python3` の結果が `/opt/homebrew/bin/python3` ならHomebrew版✅
`/usr/bin/python3` ならMac標準版なのでPATHの設定が必要です。
:::

### Gitのインストール

```bash
brew install git

# GitHubの情報を登録
git config --global user.name "GitHubのユーザー名"
git config --global user.email "GitHubのメールアドレス"
git config --global init.defaultBranch main  # ブランチ名をmainに統一（現場の慣習）

# 確認
git --version
```

初心者が混乱しやすいポイントです。

```
Git    → Macの中でのバージョン管理（ローカル）
GitHub → クラウドへのバックアップ・共有（リモート）
```

| 操作 | 意味 | ゲームで例えると |
|---|---|---|
| git init | 管理開始 | セーブ機能をONにする |
| git add | 変更を準備 | セーブするものを選ぶ |

| git push | GitHubに送る | クラウドにバックアップ |
| git pull | GitHubから取得 | 別のデバイスからロード |

### GitHubのSSH設定

#### ed25519とは？

SSH鍵を作るときの暗号化方式の種類です。

| 方式 | 特徴 |
|---|---|
| rsa | 古くから使われる定番。やや重い |
| ed25519 | 新しくて速くて安全。今の主流 ✅ |

```bash
# SSH鍵を生成
ssh-keygen -t ed25519 -C "GitHubのメールアドレス"

# 公開鍵をコピー
pbcopy < ~/.ssh/id_ed25519.pub
```

GitHubの `Settings` → `SSH and GPG keys` → `New SSH key` に貼り付けます。

```bash
# 接続テスト
ssh -T git@github.com
# Hi ユーザー名! You've successfully authenticated と表示されればOK
```

#### SSH鍵の仕組み

```
Mac（秘密鍵）←対になっている→ GitHub（公開鍵）
      ↓
自動で本人確認してくれるので毎回パスワード不要！
```

## 第2章：Django環境構築

### 仮想環境とは？

プロジェクトごとに独立したPython環境を作る仕組みです。

```
仮想環境なし → 全プロジェクトが同じライブラリを共有 → バージョン衝突のリスク😭
仮想環境あり → プロジェクトごとに独立した環境 ✅
```

### python3 -m venv venv の意味

```
python3   -m        venv          venv
  ①        ②          ③             ④

① python3  → Pythonを使う
② -m       → モジュール（機能）を実行する（moduleの略）
③ venv（1つ目） → 仮想環境を作るPython組み込み機能
④ venv（2つ目） → 作成する仮想環境フォルダの名前
```

全体の意味：「venvモジュールを使って、venvという名前のフォルダに仮想環境を作る」

### プロジェクトの作成

```bash
# プロジェクトフォルダを作成
mkdir ~/app/django-todo
cd ~/app/django-todo

# 仮想環境を作成・有効化
python3 -m venv venv
source venv/bin/activate  # (venv)と表示されればOK

# Djangoをインストール
pip install django

# プロジェクトを作成（.は現在のフォルダに作る）
django-admin startproject config .

# アプリを作成
python manage.py startapp todos
```

### startprojectとstartappの違い

![startprojectとstartappの違い](https://static.zenn.studio/user-upload/eb44229dba10-20260608.png)

```
startproject → プロジェクト全体の設定（1回だけ）
startapp     → 機能ごとのアプリ（複数作れる）
```

### settings.pyの主要設定の意味

```python
SECRET_KEY = 'xxx'  # セキュリティ用の秘密キー。本番環境では絶対に外部公開禁止
DEBUG = True        # 開発中はエラーの詳細を表示。本番では必ずFalseにする
ALLOWED_HOSTS = []  # アクセスできるドメインを指定。開発中は空でOK

INSTALLED_APPS = [...]  # Djangoで使う機能やアプリを登録。ここに書かないと認識されない

DATABASES = {...}   # 使用するDBの設定

LANGUAGE_CODE = 'ja'        # 言語設定
TIME_ZONE = 'Asia/Tokyo'    # タイムゾーン設定
```

### アプリをDjangoに登録

```python
# config/settings.py
INSTALLED_APPS = [
    ...
    'todos.apps.TodosConfig',  # フルパスで明示的に指定（現場の慣習）
]
```

:::message
省略形の`'todos'`ではなくフルパスの`'todos.apps.TodosConfig'`が現場の慣習です。
明示的に書くことでトラブルを防げます。
:::

### todosフォルダの各ファイルの意味

| ファイル | 役割 |
|---|---|
| `__init__.py` | このフォルダがPythonパッケージであることを示す。中身は空でOK |
| `admin.py` | Django管理画面にモデルを登録するファイル |
| `apps.py` | このアプリの設定ファイル。基本そのままでOK |
| `models.py` | DBのテーブル構造を定義 |
| `tests.py` | 自動テストを書くファイル |
| `views.py` | 画面の表示ロジックを書くファイル |
| `migrations/` | models.pyの変更をDBに反映するための設定が自動生成されるフォルダ |

### DjangoのMVTアーキテクチャ

Djangoはリクエストを以下の流れで処理します。

```
ブラウザ → urls.py → views.py → models.py（DB）
                         ↓
                    template.html → ブラウザに表示
```

| 役割 | ファイル | 説明 |
|---|---|---|
| Model | models.py | DBのテーブル構造を定義 |
| View | views.py | 処理ロジック |
| Template | *.html | 画面表示 |
| URL | urls.py | URLと処理の対応付け |

## 第3章：ToDoアプリの実装

### モデルの作成

```python
# todos/models.py
from django.db import models
from django.contrib.auth.models import User

class Todo(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    title = models.CharField(max_length=200)
    completed = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return self.title
```

#### 各フィールドの意味

| フィールド | 意味 |
|---|---|
| `models.Model` | これを継承することで「DBのテーブルですよ」とDjangoに伝える |
| `CharField(max_length=200)` | 最大200文字の文字列型フィールド |
| `BooleanField(default=False)` | True/Falseの2択。デフォルトはFalse（未完了） |
| `DateTimeField(auto_now_add=True)` | 作成時に日時を自動セット |
| `ForeignKey(User, on_delete=CASCADE)` | Userテーブルと紐付け。ユーザー削除時にTodoも削除 |
| `__str__` | 管理画面などで文字列表示する際に何を返すか定義 |

#### ForeignKeyとは？

2つのテーブルを関連付ける仕組みです。

```
【Todoテーブル】              【Userテーブル】
id | user_id | title          id | username
1  |    1    | 買い物    →    1  | naoki
2  |    1    | 勉強      →    1  | naoki
3  |    2    | 運動      →    2  | other_user
```

`on_delete=models.CASCADE`はユーザーが削除されたときに、そのユーザーのTodoも全部削除するという意味です。

### マイグレーション

```bash
python manage.py makemigrations  # 「こう変更しますよ」という設計書を作る
python manage.py migrate         # その設計書をDBに実際に反映する
```

:::message
マイグレーションが必要なのは**models.pyを変更したときだけ**です。
views.py・urls.py・テンプレートを変更しても不要です。
:::

### ビューの実装

#### requestとrequest.methodとは？

`request`はブラウザがサーバーに送るリクエスト情報がまるごと入ったオブジェクトです。

| プロパティ | 内容 | 例 |
|---|---|---|
| `request.method` | リクエストの種類 | 'GET' や 'POST' |
| `request.POST` | POSTで送られたデータ | {'title': '買い物'} |
| `request.user` | ログイン中のユーザー | naoki |

```
GET  → 「ページを見たい」（普通にURLにアクセス）
POST → 「データを送りたい」（フォーム送信）
```

#### @login_requiredデコレータとは？

デコレータとは関数に追加機能を付ける仕組みです。

```python
# デコレータなし（誰でもアクセスできる）
def todo_list(request):
    ...

# @login_requiredあり（未ログインはログインページへ）
@login_required
def todo_list(request):
    ...
```

```
ログイン済み → そのまま関数を実行 ✅
未ログイン   → 自動でログインページへリダイレクト ✅
```

#### get_object_or_404とは？

```python
# 通常の取得（データがなければエラーで落ちる）
Todo.objects.get(pk=pk)

# get_object_or_404（データがなければ404ページを表示）
get_object_or_404(Todo, pk=pk)
```

現場では`get_object_or_404()`を使うのが基本です。

#### todo.completed = not todo.completedとは？

`not`は真偽値を反転させる演算子です。

```
todo.completed が True  → not True  → False（未完了に戻す）
todo.completed が False → not False → True（完了にする）
```

押すたびに完了・未完了が切り替わるトグル機能です。

また`create()`と`save()`の違いも重要です。

```
Todo.objects.create() → 新規レコードを作成
todo.save()           → 既存レコードを更新
```

```python
# todos/views.py
from django.shortcuts import render, redirect, get_object_or_404
from django.contrib.auth.decorators import login_required
from .models import Todo

@login_required
def todo_list(request):
    todos = Todo.objects.filter(user=request.user)
    return render(request, 'todos/todo_list.html', {'todos': todos})

@login_required
def todo_create(request):
    if request.method == 'POST':
        title = request.POST.get('title')
        Todo.objects.create(user=request.user, title=title)
        return redirect('todo_list')
    return render(request, 'todos/todo_create.html')

@login_required
def todo_delete(request, pk):
    todo = get_object_or_404(Todo, pk=pk, user=request.user)
    if request.method == 'POST':
        todo.delete()
        return redirect('todo_list')
    return render(request, 'todos/todo_confirm_delete.html', {'todo': todo})

@login_required
def todo_complete(request, pk):
    todo = get_object_or_404(Todo, pk=pk, user=request.user)
    if request.method == 'POST':
        todo.completed = not todo.completed  # トグル
        todo.save()
        return redirect('todo_list')
```

### URL設定

#### URLの`<int:pk>`とは？

URLの中に整数を埋め込む仕組みです。この数字がそのままviews.pyのpk引数に渡されます。

```
delete/<int:pk>/

delete/1/ → pk=1
delete/2/ → pk=2
delete/3/ → pk=3
```

```python
# todos/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path('', views.todo_list, name='todo_list'),
    path('create/', views.todo_create, name='todo_create'),
    path('delete/<int:pk>/', views.todo_delete, name='todo_delete'),
    path('complete/<int:pk>/', views.todo_complete, name='todo_complete'),
]
```

### テンプレートの{% csrf_token %}とは？

CSRF（クロスサイトリクエストフォージェリ）攻撃を防ぐDjangoの仕組みです。

```
【悪意のあるサイトからの攻撃の流れ】
① ユーザーが正規サイトにログイン済み
② 悪意のあるサイトを開く
③ 裏で正規サイトに不正なPOSTリクエストを送る
④ 正規サイトが本人のリクエストと勘違いして処理してしまう😱

【csrf_tokenで防ぐ】
正規フォーム → トークンあり ✅ → 処理する
悪意のサイト → トークンなし ❌ → 拒否する
```

DjangoはデフォルトでCSRF対策が有効になっています。**POSTフォームには必ず書く**のが現場ルールです。

## 第5章：Docker + PostgreSQL

### Dockerとは？

アプリをどこでも同じように動かすための「箱（コンテナ）」を作るツールです。

```
【Dockerなし】
「自分のMacでは動くのに、本番サーバーでは動かない！」← よくある問題😭

【Dockerあり】
アプリの動作に必要なもの（OS・ライブラリ・設定）をコンテナに詰め込む
→ どこでも同じように動く ✅
```

### Dockerfileの各命令の意味

| 命令 | 意味 |
|---|---|
| `FROM` | コンテナの土台となるベースイメージを指定 |
| `WORKDIR` | コンテナ内での作業フォルダを設定 |
| `COPY` | ファイルをコンテナにコピー |
| `RUN` | コマンドを実行（イメージビルド時） |
| `CMD` | コンテナ起動時に実行するコマンド |
| `slim` | 必要最低限のパッケージだけ入った軽量版 |
| `0.0.0.0` | コンテナ外からのアクセスを許可（127.0.0.1は自分のMacからのみ） |

### Dockerfile

```dockerfile
FROM python:3.14-slim    # Python入りの軽量Linux
WORKDIR /app             # 作業フォルダを/appに設定
COPY requirements.txt .  # 依存関係ファイルをコピー
RUN pip install -r requirements.txt  # ライブラリをインストール
COPY . .                 # ソースコードを全てコピー
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]  # サーバー起動
```

### docker-compose.ymlの各設定の意味

```yaml
services:
  web:
    build: .              # 現在のフォルダのDockerfileでビルド
    ports:
      - "8000:8000"       # Mac側:コンテナ側のポートを紐付け
    depends_on:
      - db                # dbコンテナが起動してからwebを起動
    env_file:
      - .env              # .envから環境変数を読み込む

  db:
    image: postgres:16    # PostgreSQL16のイメージを使用
    volumes:
      - postgres_data:/var/lib/postgresql/data  # データを永続化
    env_file:
      - .env

volumes:
  postgres_data:          # データ永続化用のボリューム
```

### なぜSQLiteからPostgreSQLに切り替えるのか

| | SQLite | PostgreSQL |
|---|---|---|
| 設定 | 不要（ファイル1つ） | Docker設定が必要 |
| 本番環境での使用 | △ | ◎ |
| Djangoとの相性 | ○ | ◎ |
| 学習順序 | 最初 | 後から |

### .env（機密情報の管理）

```
SECRET_KEY=your-secret-key
DEBUG=True
POSTGRES_DB=django_todo
POSTGRES_USER=django_user
POSTGRES_PASSWORD=django_password
```

:::message alert
`.env`は絶対にGitHubに上げてはいけません！
`.gitignore`に必ず追加してください。
チームメンバーへの案内として`.env.example`を作るのが現場の慣習です。
:::

### 起動コマンド

```bash

docker compose exec web python manage.py migrate             # マイグレーション
docker compose exec web python manage.py createsuperuser     # 管理者ユーザー作成
```

### ⑧ ForeignKey追加時のマイグレーション警告

**エラー内容**

```
It is impossible to add a non-nullable field 'user' to todo
without specifying a default.
Please select a fix:
 1) Provide a one-off default now
 2) Quit and manually define a default value
```

**原因**

既存のTodoレコードに`user`フィールドが存在しないため、デフォルト値が必要になった。

**解決方法**

`1`を選択し、デフォルト値として`1`（管理者ユーザーのID）を入力する。

:::message
新規プロジェクトでは最初からuserフィールドを設計しておくことが理想。
:::
