## はじめに

前回の記事でAWSデプロイの全体構成を整理しました。今回はその最初のステップとして、**RDS（PostgreSQL）の作成**を実際にやってみました。

AWSコンソールの操作を一つひとつスクリーンショットで確認しながら進めたので、同じようにやれば誰でもできるようにまとめています。

---

## RDSとは？

RDS（Relational Database Service）は**AWSが管理するクラウド上のデータベース**です。

ローカル開発では `docker-compose.yml` の `db` コンテナがDBの役割を担っていましたが、本番環境ではそれをRDSに置き換えます。

```
【ローカル】
docker-compose.yml
  └── db（PostgreSQL 16コンテナ）← 自分のMacで動く

【本番（AWS）】
RDS（PostgreSQL 16）← AWSが管理・運用してくれる
```

RDSの特徴はこちらです。

| 特徴 | 内容 |
|---|---|
| 自動バックアップ | 毎日自動でバックアップしてくれる |
| 自動パッチ適用 | セキュリティアップデートを自動適用 |
| サーバー管理不要 | OS・DBのメンテナンスはAWSにおまかせ |
| 高可用性 | マルチAZ構成で冗長化も可能（今回は不使用）|

## 今回の構成

まなびクエストのRDS設定はこちらです。

| 項目 | 設定値 | 理由 |
|---|---|---|
| エンジン | PostgreSQL 16.14 | ローカルと同じバージョン |
| インスタンスクラス | db.t3.micro | 最小・最安値 |
| ストレージ | 20GiB（gp2） | 開発段階では十分 |
| パブリックアクセス | なし | セキュリティのため |
| DB名 | manabi_quest | ローカルの.envと同じ |
| ユーザー名 | postgres | ローカルの.envと同じ |

## 作成手順

### ① RDSダッシュボードを開く

AWSコンソールの検索バーに `RDS` と入力して「Aurora and RDS」を選択します。

「フル設定で作成」の**「作成」ボタン**をクリックします。

---

### ② エンジンの選択

エンジンのタイプで **「PostgreSQL」** を選択します。

:::message alert
「Aurora (PostgreSQL Compatible)」は似ていますが別物です。Auroraは高機能・高コスト（月$50〜）なので、通常のPostgreSQLを選びましょう。
:::

作成方法は**「フル設定」**を選択します。

---

### ③ テンプレートの選択

**「開発/テスト」** を選択します。

| テンプレート | 費用 | 用途 |
|---|---|---|
| 本番稼働用 | 高い（マルチAZ） | 本番サービス向け |
| **開発/テスト** | **安い** | **← 今回はこれ** |
| サンドボックス | 安い | 学習用 |

---

### ④ 可用性と耐久性

**「シングルAZ DBインスタンスデプロイ（1インスタンス）」** を選択します。

マルチAZは複数のアベイラビリティーゾーンにDBを冗長化する機能ですが、コストが2〜3倍になります。開発段階では不要です。

---

### ⑤ 設定（DBの名前・ユーザー・パスワード）

以下の通りに入力します。

| 項目 | 値 |
|---|---|
| エンジンバージョン | PostgreSQL 16.14 |
| RDS延長サポート | **OFFにする**（有料になるため） |
| DBインスタンス識別子 | manabi-quest-db |
| マスターユーザー名 | postgres |
| 認証情報管理 | セルフマネージド |
| パスワード | ローカルの.envのDB_PASSWORDと同じ値 |

:::message
**エンジンバージョンについて**
PostgreSQL 16.xを選ぶとRDS延長サポートのチェックが自動でONになる場合があります。これは有料なので必ずOFFにしてください。
:::

---

### ⑥ インスタンスクラスの選択

「バースト可能クラス（tクラスを含む）」を選択して、インスタンスタイプを **`db.t3.micro`** に変更します。

```
db.m7g.large（デフォルト）→ 約$100〜/月  ← 高い！
db.t3.micro（変更後）     → 約$15〜/月   ← 開発用はこれ
```

---

### ⑦ ストレージの設定

| 項目 | 設定値 |
|---|---|
| ストレージタイプ | 汎用SSD（gp2） |
| ストレージ割り当て | 20 GiB |
| ストレージの自動スケーリング | **OFFにする** |

自動スケーリングをONにすると、使用量に応じてストレージが自動で増えてその分課金されます。開発中は20GiBで十分なのでOFFにします。

---

### ⑧ 接続の設定

| 項目 | 設定値 | 理由 |
|---|---|---|
| コンピューティングリソース | EC2に接続しない | 後で手動設定 |
| VPC | Default VPC | そのまま |
| パブリックアクセス | **なし** | セキュリティのため |
| VPCセキュリティグループ | 新規作成 | manabi-quest-rds-sg |

**パブリックアクセスを「なし」にする理由**

```
パブリックアクセス「あり」
　→ インターネットから直接DBに接続できる
　→ 攻撃を受けるリスクがある ❌

パブリックアクセス「なし」← こちらを選択
　→ VPC内（ECSコンテナ）からしか接続できない
　→ 安全！ ✅
```

---

### ⑨ モニタリングの設定

以下をすべてOFFにします（追加料金が発生するため）。

- Performance Insights → **OFF**
- 拡張モニタリング → **OFF**
- DevOps Guru → **OFF**

---

### ⑩ 追加設定

**「最初のデータベース名」** に `manabi_quest` を入力します。

:::message alert
ここを空白のままにすると、DBインスタンスは作られますがDB（スキーマ）が作られません。Djangoが接続できなくなるので必ず入力してください。
:::

その他の設定です。

| 項目 | 設定値 |
|---|---|
| バックアップ保持期間 | **1日**（デフォルトの7日は無料枠超過） |
| 削除保護 | OFF（開発中は削除しやすくする） |

---

### ⑪ データベースの作成

「データベースの作成」ボタンをクリックします。

作成には**約10〜15分**かかります。ステータスが「作成中」→「利用可能」になれば完了です。

---

## 作成完了後にやること

ステータスが「利用可能」になったら、**エンドポイント**を確認します。

RDSの詳細画面 → 「接続とセキュリティ」タブ → エンドポイントをコピー

```
例: manabi-quest-db.cf00uqi8kwtd.ap-northeast-1.rds.amazonaws.com
```

このエンドポイントを `DB_HOST` としてDjangoの設定（`.env`）に追加します。

```bash
# .env（本番用）
DB_HOST=manabi-quest-db.cf00uqi8kwtd.ap-northeast-1.rds.amazonaws.com
DB_PORT=5432
DB_NAME=manabi_quest
DB_USER=postgres
DB_PASSWORD=（設定したパスワード）
```

---

## コスト削減のポイント

開発・テスト中はRDSを**使うときだけ起動**するのがコスパ良いです。

```
使わないとき
　→ スナップショットを作成してDBを削除
　→ 月$0

使うとき
　→ スナップショットからDBを復元
　→ 作業中のみ課金（時間単位）
```

常時起動だと月$15〜25かかりますが、週数回の作業なら月数百円に抑えられます。
